# learn from vibe-coding documentation

## 0. 文档说明

本文档用于指导开发一个用于个人 vibe coding 项目技术沉淀的 Skill。

这个 Skill 的目标不是做项目管理，不是写作品集，不是评估技术强弱，也不是建立完整的前端学习路线。它只负责在每个项目结束后，抽取项目中使用过的主要技术模块，并将这些技术模块沉淀到一个可长期维护、可视化展示的「技术岛屿地图」中。

Skill 的核心产物包括：

1. 一套轻量的数据结构。
2. 一套 Notion 存储方案。
3. 一个只读 HTML 技术岛屿地图页面。
4. 一套项目复盘时的技术抽取规则。
5. 一套新增技术点进入地图的判断规则。

当前版本只做技术沉淀，不包含以下内容：

- 不沉淀产品背景。
- 不沉淀用户场景。
- 不沉淀作品集表达。
- 不沉淀视觉规范。
- 不做能力强弱评估。
- 不做学习建议。
- 不做技术补足提醒。
- 不做复杂统计。
- 不做编辑功能。
- 不做筛选功能。

---

## 1. Skill 定位

### 1.1 Skill 名称

建议名称：

**learn from vibe-coding**

### 1.2 核心目标

帮助用户在完成一个 vibe coding 项目后，快速沉淀该项目中涉及的主要技术模块，并将这些技术模块更新到一张全局「技术岛屿地图」中。

地图用于回答一个问题：

**在我做过的 vibe coding 项目中，我已经实践过哪些主要技术点，它们分别属于哪类技术领域。**

### 1.3 核心原则

Skill 必须遵守以下原则：

1. **只沉淀技术，不沉淀产品内容。**
   
   项目目标、用户问题、作品集表达、业务价值不进入本 Skill。

2. **只记录主要技术模块，不拆到小功能。**
   
   不记录按钮点击、弹窗显示、某个小交互动效等细碎功能。只记录项目中有代表性的技术能力，例如 Canvas、Three.js、Vite、localStorage、Web Audio、API、Git、Prompt 拆解等。

3. **只区分使用和未使用。**
   
   不设置复杂掌握等级，不设置“接触 / 使用 / 核心 / 可复用”等多档深度。当前版本只判断该技术点是否在项目中被实际使用过。

4. **地图只做展示，不做编辑。**
   
   数据编辑和维护在 Notion 中完成。HTML 页面只读取数据并展示。

5. **Notion 是默认存储，不是唯一存储。**
   
   系统应通过统一数据结构收口，后续可以替换为飞书、Airtable、本地 JSON、Obsidian 等其他存储方式。

6. **页面是岛屿式全景图，不是表格后台。**
   
   页面主体应平铺展示 6 类技术群岛，每个群岛里展示对应技术点，点亮表示已使用，灰色表示未使用。

## 2. 使用场景

### 2.1 单项目结束后的技术复盘

用户完成一个 vibe coding 项目后，向 Skill 提供以下材料中的一种或多种：

- 项目名称。
- 项目链接。
- 项目代码文件。
- 项目 README。
- 项目说明。
- 用户对项目中技术使用情况的描述。
- 与 AI 协作过程中的关键问题或弯路。

Skill 根据材料抽取该项目涉及的主要技术模块，并生成适合写入 Notion 的结构化内容。

### 2.2 全局技术岛屿地图更新

当新增项目复盘后，Skill 需要判断：

- 是否出现了已有技术点。
- 是否出现了新的技术点。
- 新技术点应该归入哪个技术群岛。
- 新技术点是否应该进入主地图。
- 已使用的技术点是否需要点亮。

### 2.3 HTML 页面展示

HTML 页面读取统一数据后，展示 6 个技术群岛。

每个群岛内包含若干技术点。技术点只有两种状态：

- 未使用：灰色或低透明度。
- 已使用：点亮或高亮。

用户点击技术点后，页面展示该技术点相关的项目应用记录，包括：

- 出现在哪些项目中。
- 在项目里的作用。
- 弯路与修正。
- 是否可复用。
- 备注。

## 3. 信息架构

当前版本只保留 3 类核心数据。

### 3.1 项目

项目是技术沉淀的来源。

每个项目只作为索引，不记录产品背景和作品集内容。

项目数据用于回答：

**这个技术点出现在哪些项目中。**

### 3.2 技术点

技术点是地图上的节点。

技术点可以是具体技术、工具、框架、能力类型或 AI 协作方法。

例如：

- HTML
- CSS
- Canvas
- SVG
- Three.js
- Web Audio
- JSON
- Mock 数据
- localStorage
- API
- npm
- Vite
- Git
- 静态部署
- Prompt 拆解
- 上下文文件
- Bug 修复
- 代码审查

技术点用于回答：

**我在技术地图上有哪些节点，哪些已经点亮。**

### 3.3 项目-技术应用

项目-技术应用是最核心的数据。

它表示：

**某个项目中使用了某个技术点，并记录这个技术点在该项目中的作用、AI 协作、弯路和复用判断。**

这张表是连接项目和技术点的关系表。

## 4. 技术群岛分类

HTML 页面平铺 6 类技术群岛。

每一类技术对应一个群岛，每个群岛里放若干技术点。

### 4.1 页面结构岛

用于承载页面搭建、布局、基础 UI 结构相关技术。

建议初始技术点：

- HTML
- CSS
- 布局
- 响应式
- 表单
- 弹窗
- 组件结构
- 页面路由

说明：

页面结构岛关注的是页面能否被组织出来，包括结构、布局、基础样式、响应式适配、表单与弹窗等通用页面能力。

### 4.2 交互状态岛

用于承载用户操作和页面状态变化相关技术。

建议初始技术点：

- 点击事件
- 拖拽
- 上传
- 状态切换
- Loading
- 结果态
- 错误态
- 键盘 / 鼠标 / 触摸事件

说明：

交互状态岛关注的是用户行为如何触发页面变化，包括用户点击、上传、拖拽、输入、状态切换、加载态、结果态、错误态等。

### 4.3 动画表现岛

用于承载视觉表现、图形绘制、动画渲染相关技术。

建议初始技术点：

- CSS 动画
- Canvas
- SVG
- Three.js
- requestAnimationFrame
- 图层叠加
- 粒子效果
- 过渡动效

说明：

动画表现岛关注的是视觉层面的动态表现，包括普通 CSS 动效、Canvas 绘制、SVG 图形、Three.js 三维渲染、图层叠加、粒子效果和动画循环等。

### 4.4 数据处理岛

用于承载数据组织、数据读取、前端状态数据、轻量存储相关技术。

建议初始技术点：

- JSON
- Mock 数据
- 列表渲染
- 筛选排序
- localStorage
- API 数据
- 数据格式转换
- 前端状态数据

说明：

数据处理岛关注的是页面如何使用数据，包括 mock 数据、JSON 结构、列表渲染、筛选排序、本地存储、API 数据读取和数据格式转换等。

### 4.5 工程部署岛

用于承载项目运行、构建、版本管理、资源管理、发布相关技术。

建议初始技术点：

- npm
- Vite
- Git
- 静态资源管理
- 打包
- Vercel
- GitHub Pages
- 环境变量

说明：

工程部署岛关注的是项目如何跑起来、如何管理、如何发布，包括依赖安装、开发服务器、打包、Git 版本管理、静态资源管理、部署平台和环境配置等。

### 4.6 AI 协作岛

用于承载 vibe coding 过程中的 AI 协作方法。

建议初始技术点：

- 需求拆解
- 上下文文件
- 分步提示
- Bug 修复
- 代码审查
- 重构提示
- 多轮迭代
- Agent 协作

说明：

AI 协作岛关注的是用户如何与 AI 一起完成开发，包括如何拆任务、如何提供上下文、如何分步生成、如何让 AI 修 bug、如何审查 AI 代码、如何推动重构和多轮迭代。

## 5. 数据字段设计

### 5.1 项目库字段

项目库只记录项目索引信息。

| 字段名 | 字段类型 | 是否必填 | 说明 |
|---|---|---|---|
| 项目名称 | Title | 必填 | 项目的名称，例如「互动写字页 Demo」「网格小游戏 Demo」「武林舆图」 |
| 项目链接 / 文件路径 | URL / Text | 选填 | 项目线上地址、GitHub 地址、本地文件路径或代码仓库位置 |
| 复盘时间 | Date | 选填 | 完成技术沉淀的时间 |

项目库不记录用户问题、产品目标、作品集价值、业务背景等内容。

### 5.2 技术点库字段

技术点库记录地图上的节点。

| 字段名 | 字段类型 | 是否必填 | 说明 |
|---|---|---|---|
| 技术名称 | Title | 必填 | 技术点名称，例如 Canvas、Three.js、Vite、Git、Prompt 拆解 |
| 所属技术岛屿 | Select | 必填 | 页面结构岛、交互状态岛、动画表现岛、数据处理岛、工程部署岛、AI 协作岛 |
| 技术简介 | Text | 必填 | 用简短语言解释这个技术点是什么，主要解决什么问题 |
| 是否使用 | Checkbox / Select | 必填 | 只保留两档：未使用、已使用 |

字段说明：

- 「技术简介」必须保留，用于用户点击节点后快速理解该技术点。
- 「是否使用」不表示掌握程度，只表示是否已经在真实项目中使用过。
- 不设置使用深度、掌握等级、优先级、薄弱程度等字段。

### 5.3 项目-技术应用库字段

项目-技术应用库是核心表。

| 字段名 | 字段类型 | 是否必填 | 说明 |
|---|---|---|---|
| 项目 | Relation | 必填 | 关联项目库中的项目 |
| 技术点 | Relation | 必填 | 关联技术点库中的技术点 |
| 在项目里的作用 | Text | 必填 | 该技术点在该项目中承担什么作用 |
| 弯路与修正 | Text | 选填 | 是否走过明显弯路，后来如何修正 |
| 是否可复用 | Checkbox / Select | 选填 | 该技术点在这个项目中的应用是否值得后续复用 |
| 备注 | Text | 选填 | 可记录代码位置、文件名、补充说明等 |

字段说明：

- 「在项目里的作用」是必填字段，因为它说明该技术点为什么会被点亮。
- 「弯路与修正」只记录重要弯路，不记录小 bug。
- 「是否可复用」只记录这个应用经验是否有复用价值，不代表用户已经掌握该技术。

## 6. 点亮规则

### 6.1 点亮含义

技术点被点亮，只表示：

**该技术点已经在至少一个项目中被实际使用过。**

技术点未点亮，表示：

**该技术点尚未在记录过的项目中使用。**

点亮不表示：

- 不表示已经掌握。
- 不表示已经熟练。
- 不表示能力强。
- 不表示完成学习。
- 不表示必须补齐未点亮节点。

### 6.2 点亮条件

只要满足以下条件，技术点即可被标记为「已使用」：

1. 该技术点在项目中承担了明确作用。
2. 项目-技术应用库中存在该技术点与某个项目的关联记录。
3. 「在项目里的作用」字段能够说明它解决了什么问题。

### 6.3 不点亮条件

以下情况不应点亮：

1. 只是讨论过，但没有实际用于项目。
2. 只是用户听说过或计划未来使用。
3. 只是某个依赖内部使用，用户没有直接接触。
4. 只是 AI 生成代码中偶然出现，但用户没有确认其作用。
5. 无法说清该技术点在项目中承担什么作用。

### 6.4 是否使用字段的更新

当某个技术点在任意项目中出现有效应用记录时，技术点库中的「是否使用」应更新为「已使用」。

当某个技术点没有任何项目应用记录时，技术点库中的「是否使用」应保持「未使用」。

## 7. 新增技术点规则

### 7.1 新技术点的来源

新技术点可能来自：

- 用户主动说明。
- 项目代码分析。
- README 或 PRD 中提到。
- AI 协作记录中出现。
- 复盘时用户补充。

### 7.2 新技术点是否进入地图

不是所有技术名词都必须进入地图。

进入地图的技术点应满足至少一个条件：

1. 它是项目中明确使用的主要技术模块。
2. 它代表一种可复用的技术能力。
3. 它属于 6 个技术群岛中的稳定技术点。
4. 它未来可能在多个 vibe coding 项目中重复出现。

不建议进入地图的内容：

1. 一次性小插件。
2. 很细碎的属性或 API。
3. 某个库内部自动处理的能力。
4. 用户没有直接理解或接触的底层实现。
5. 无法归类到 6 个技术群岛的临时名词。

### 7.3 新技术点处理方式

当 Skill 发现新技术点时，不应直接强行写入主地图。应先输出建议：

- 新技术点名称。
- 建议归属岛屿。
- 是否建议进入地图。
- 如果不建议进入地图，说明可放在备注或项目记录中。

用户确认后再加入技术点库。

### 7.4 每个群岛的节点数量

为了保持地图可读性，每个群岛第一版建议控制在 8-12 个主要技术点。

如果后续技术点过多，应优先合并为更大的主节点，而不是无限扩展地图。

示例：

- 不要把「鼠标按下」「鼠标移动」「鼠标松开」分别做成节点。
- 可以统一归为「鼠标 / 触摸事件」。

## 8. Notion 存储方案

### 8.1 Notion 的角色

Notion 是当前版本的主数据存储工具。

Notion 负责：

- 存项目。
- 存技术点。
- 存项目-技术应用记录。
- 让用户手动查看和维护数据。

Notion 不负责：

- 不负责做岛屿地图可视化。
- 不负责做复杂统计。
- 不负责做页面展示。
- 不负责前端交互。

### 8.2 数据库数量

Notion 中建立 3 张数据库：

1. 项目库。
2. 技术点库。
3. 项目-技术应用库。

### 8.3 数据关系

项目库与项目-技术应用库是一对多关系。

一个项目可以关联多个项目-技术应用记录。

技术点库与项目-技术应用库是一对多关系。

一个技术点可以出现在多个项目-技术应用记录中。

关系结构：

```text
项目库
  ↓
项目-技术应用库
  ↑
技术点库
```

### 8.4 数据维护方式

第一版推荐手动或半自动维护。

流程：

1. 用户完成项目。
2. 用户触发 Skill 进行技术复盘。
3. Skill 输出结构化复盘内容。
4. 用户将内容写入 Notion，或由后续自动化工具写入 Notion。
5. HTML 页面读取导出的统一 JSON 进行展示。

### 8.5 数据导出方式

HTML 页面不直接依赖 Notion 页面展示。

推荐链路：

```text
Notion 数据库
↓
导出 / 同步
↓
统一 JSON 数据
↓
HTML 页面读取 JSON
```

这样可以保证：

- 前端页面不绑定 Notion。
- 后续可替换为其他存储工具。
- 开源用户可以使用本地 JSON 直接运行。

## 9. 统一数据源要求

### 9.1 数据源定位

HTML 页面只读取统一数据源。

统一数据源可以由 Notion 导出，也可以由其他工具生成。

当前建议统一为 JSON 文件。

### 9.2 数据源必须包含的信息

统一数据源至少要包含：

1. 项目列表。
2. 技术点列表。
3. 项目-技术应用列表。

### 9.3 数据源与 HTML 的关系

HTML 页面不关心数据从哪里来。

HTML 页面只关心：

- 有哪些技术群岛。
- 每个群岛有哪些技术点。
- 每个技术点是否已使用。
- 点击技术点后有哪些项目应用记录。

## 10. HTML 页面设计要求

### 10.1 页面类型

页面是一个只读展示页面。

不做编辑。

不做筛选。

不做复杂统计。

不做后台管理。

### 10.2 页面主结构

页面分为 3 个区域：

1. 顶部标题区。
2. 技术群岛地图区。
3. 技术点详情区。

### 10.3 顶部标题区

顶部只展示标题。

建议标题：

**learn from vibe-coding**

顶部不展示：

- 不展示更新时间。
- 不展示数据来源。
- 不展示项目数。
- 不展示技术点数。
- 不展示使用比例。
- 不展示薄弱项。
- 不展示学习建议。

### 10.4 技术群岛地图区

主体区域平铺 6 个群岛。

每个群岛对应一类技术：

1. 页面结构岛。
2. 交互状态岛。
3. 动画表现岛。
4. 数据处理岛。
5. 工程部署岛。
6. AI 协作岛。

每个群岛内部展示技术点节点。

技术点节点只展示技术名称。

节点状态只有两种：

- 未使用。
- 已使用。

### 10.5 岛屿视觉形态

每个群岛应该有独立区域感。

建议视觉表达：

- 每个群岛是一个柔和的不规则区域或圆角区域。
- 群岛背景使用浅色块区分。
- 群岛标题位于区域顶部。
- 技术点以小圆点、胶囊标签或小岛礁形式平铺在群岛内部。
- 已使用节点使用高亮色。
- 未使用节点使用灰色或低透明度。

不建议：

- 不做地理地图。
- 不做中国地图形态。
- 不做复杂地铁线。
- 不做大量连线。
- 不做密集卡片墙。
- 不做表格主视图。

### 10.6 技术点详情区

点击某个技术点后，展示详情。

详情可以在右侧面板或底部面板中出现。

详情内容包括：

- 技术名称。
- 所属技术岛屿。
- 技术简介。
- 是否使用。
- 出现过的项目。
- 在项目里的作用。
- 弯路与修正。
- 是否可复用。
- 备注。

如果某个技术点未使用，详情区只展示：

- 技术名称。
- 所属技术岛屿。
- 技术简介。
- 当前状态：未使用。

不展示空项目记录。

### 10.7 默认状态

页面首次打开时：

- 展示完整 6 个技术群岛。
- 不默认展开任何详情。
- 可在详情区展示一句提示：点击一个技术点查看项目应用记录。

### 10.8 交互要求

HTML 页面只需要以下交互：

1. 点击技术点。
2. 展示技术点详情。
3. 已使用 / 未使用状态视觉区分。

不需要：

- 不需要搜索。
- 不需要筛选。
- 不需要排序。
- 不需要编辑。
- 不需要新增。
- 不需要删除。
- 不需要登录。
- 不需要权限管理。

## 11. Skill 工作流

### 11.1 触发方式

用户可以用类似指令触发：

- 复盘这个项目的技术沉淀。
- 分析这个项目用了哪些技术模块。
- 把这个项目更新到技术岛屿地图。
- 根据这份代码生成技术沉淀记录。
- 帮我整理这个 vibe coding 项目的技术使用和弯路。

### 11.2 输入材料

Skill 可接受以下输入：

- 项目名称。
- 项目地址。
- 项目说明。
- 代码文件。
- README。
- 用户补充说明。
- AI 协作过程描述。
- 技术弯路描述。

### 11.3 分析步骤

Skill 分析项目时按以下步骤执行：

1. 识别项目中的主要技术模块。
2. 将技术模块归入 6 个技术群岛。
3. 判断技术点是否已存在。
4. 对每个主要技术点，提炼其在项目里的作用。
5. 提炼明显弯路与修正。
6. 判断是否可复用。
7. 输出可写入 Notion 的结构化内容。
8. 如发现新技术点，输出新增建议，由用户确认。

### 11.4 输出内容

Skill 输出应包含三部分：

第一部分：项目记录。

第二部分：技术点更新。

第三部分：项目-技术应用记录。

如有新技术点，额外输出：

第四部分：待确认新增技术点。

### 11.5 输出边界

Skill 不输出：

- 不输出产品分析。
- 不输出用户洞察。
- 不输出作品集文案。
- 不输出学习计划。
- 不输出补课建议。
- 不输出能力评价。
- 不输出技术强弱判断。

## 15. HTML 页面视觉与布局规范

### 15.1 页面气质

页面应像一张轻量、清晰、有探索感的技术岛屿地图。

关键词：

- 清爽。
- 平铺。
- 分区明确。
- 点亮感。
- 不后台化。
- 不课程表化。
- 不复杂统计化。

### 15.2 页面布局

建议使用单页布局。

结构：

```text
顶部标题

6 个技术群岛平铺区域

点击节点后的详情面板
```

可以采用两种布局：

方案 A：详情面板固定在右侧。

适合桌面端。

```text
左侧 / 中间：技术群岛地图
右侧：技术点详情
```

方案 B：详情面板在底部弹出。

适合移动端或窄屏。

```text
上方：技术群岛地图
下方：技术点详情
```

### 15.3 群岛布局

6 个群岛可以采用 2 行 3 列平铺。

桌面端：

```text
页面结构岛    交互状态岛    动画表现岛
数据处理岛    工程部署岛    AI 协作岛
```

移动端：

```text
页面结构岛
交互状态岛
动画表现岛
数据处理岛
工程部署岛
AI 协作岛
```

### 15.4 节点样式

技术点节点建议使用小胶囊、小圆点或小岛礁形态。

节点只展示技术名称。

状态：

- 未使用：灰色、低透明度、弱边框。
- 已使用：高亮色、清晰边框或轻微发光。

不在节点上展示：

- 不展示项目数量。
- 不展示证据。
- 不展示说明。
- 不展示使用等级。
- 不展示图标。

### 15.5 详情面板样式

详情面板应轻量，不要压过地图主体。

内容分组：

1. 技术点基本信息。
2. 项目应用记录。
3. 弯路与修正。
4. 复用判断与备注。

如果项目应用记录较多，可以用列表卡片展示。

## 18. 质量验收标准

### 18.1 数据验收

数据层应满足：

- 每个项目都能关联多个技术点。
- 每个技术点都归属于一个技术群岛。
- 每个技术点都有技术简介。
- 每个项目-技术应用记录都说明技术点在项目里的作用。
- 已使用状态能根据项目应用记录被正确展示。

页面应满足：

- 页面打开后能看到 6 个技术群岛。
- 每个群岛内能看到技术点。
- 已使用和未使用节点视觉差异明显。

- 页面不显得像后台表格。

- 只输出技术沉淀内容。
- 不扩展到产品分析。
- 不输出作品集表达。

- 每个项目只抽取主要技术模块。
- 新技术点需要给出归类建议。

## 19. 典型错误与禁止项

### 19.1 禁止把系统做成学习打卡

不要出现：

- 今日学习。
- 待补足。
- 薄弱能力。
- 学习路线。
- 推荐课程。
- 技术掌握百分比。

### 19.2 禁止把系统做成作品集管理

不要出现：

- 用户洞察。
- 产品价值。
- 作品集亮点。
- 面试表达。
- 项目故事。
- 业务背景。

### 19.3 禁止把系统做成完整项目管理工具

不要出现：

- 任务进度。
- Todo。
- 需求排期。
- Bug 管理。
- 项目成员。
- 迭代计划。

### 19.4 禁止技术点拆得过细

不要把以下内容单独做成地图节点：

- 某个按钮点击。
- 某个弹窗打开。
- 某个 CSS 属性。
- 某个变量。
- 某个一次性函数。
- 某个小 bug。

应优先归并到更大的技术点中。

## 20. 最终方案摘要

当前 Skill 的最终方案为：

**一个用于 vibe coding 项目技术沉淀的轻量 Skill。**

它通过项目复盘抽取主要技术模块，记录技术点在项目中的作用、弯路与修正、是否可复用，并将数据存储到 Notion 的 3 张数据库中。

HTML 页面只负责读取统一数据源，并以 6 个平铺技术群岛的方式展示技术点。技术点只有未使用和已使用两种状态。点击技术点后展示详情。

该系统不做学习评估，不做作品集表达，不做产品分析，不做复杂统计，不做筛选，不做编辑。

它只回答一个核心问题：

**我在自己的 vibe coding 项目中，已经实际使用过哪些技术点。**
